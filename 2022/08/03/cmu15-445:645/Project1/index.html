TASK #1 - LRU REPLACEMENT POLICY

LRU 算法策略：最近最久未用

实现LRU替换策略来维护 buffer pool 中的页面使用情况，在src/include/buffer/lru_replacer.h 中实现一个名为 LRUReplacer 的新子类，并在 src/buffer/lru_replacer.cpp中完成其实现。

LRUReplacer 的最大页面数和 buffer pool的大小相同，因为它包含了BufferPoolManager的所有帧，然而在给定的任一时刻，所有的帧并不一定都在LRUReplacer中，初始化时，没有帧在其中。只有那些新被 unpin 的页面才在其中。

需要实现几个方法：

  • bool Victim(frame_id_t*)：删除 最近没有访问的页面，将其内容存储在输入参数中并返回 True ，如果 Replacer 是空的，则返回 False。
  • Pin (frame_id_t)：在BufferPoolManager中一个页面被pin从而成为帧时，才调用此方法，从LRUReplacer中移除此页。
  • Unpin(frame_id_t)：这个方法只有当页面的pin_count变成0后才会被调用，这个方法应该将包含未固定页面的帧添加到 LRUReplacer。
  • Size() ：返回 LRUReplacer 的页面大小。

可以使用STL内置容器，可以假设不超内存但是确保操作是线程安全的。

只需实现LRU策略，不必管项目中的clock置换策略。

TASK #2 - BUFFER POOL MANAGER INSTANCE

实现缓冲池管理器(BufferPoolManager)。BufferPoolManagerInstance负责从DiskManager获取数据库页面并将它们存储在内存中。BufferPoolManagerInstance还可以在要求它把脏页写入磁盘，或当它需要驱逐一个页以便为新页腾出空间时，将脏页写入磁盘。

为了确保您的实现能够正确地与系统的其余部分一起工作，我们将为您提供一些已经填写好的功能。您也不需要实现实际读写数据到磁盘的代码(即DiskManager)。我们将为您提供这一功能。

系统中的所有内存页面均由Page对象表示。BufferPoolManagerInstance不需要了解这些页面的内容。 但是，作为系统开发人员，重要的是要了解Page对象只是一个容器，它用于存储内存的容器，因此并不特定于唯一页面。 也就是说，每个Page对象都包含一块内存，DiskManager会将其作为用来「复制从磁盘读取的物理页面内容」的地点。 BufferPoolManagerInstance将在将其来回移动到磁盘时重用相同的Page对象来存储数据。 这意味着在系统的整个生命周期中，相同的Page对象可能包含不同的物理页面。Page对象的标识符page_id可以跟踪其包含的物理页面。 如果Page对象不包含物理页面即磁盘上的页面，则必须将其page_id设置为INVALID_PAGE_ID。

每个Page对象还维护一个计数器，以显示“固定”该页面的线程数。BufferPoolManagerInstance不许释放已被pinned 的页面，每个Page对象还跟踪其脏标记。您的工作是判断页面在解除固定之前是否已经被修改（若已经被修改，则把脏标记置为1）。BufferPoolManagerInstance必须将脏页的内容写回磁盘，然后才能重用该对象。

BufferPoolManager的实现将使用在此分配的前面步骤中创建的LRUReplacer类。它将使用LRUReplacer来跟踪何时访问页对象，以便在必须释放一个帧以为从磁盘复制新的物理页腾出空间时，它可以决定取消哪个页对象。

需要实现在(src/buffer/buffer_pool_manager.cpp)的以下函数 - FetchPageImpl(page_id) - NewPageImpl(page_id) - UnpinPageImpl(page_id, is_dirty) - FlushPageImpl(page_id) - DeletePageImpl(page_id) - FlushAllPagesImpl()

FetchPgImp，如果空闲链表中没有可用的页面，并且当前所有其他页面都被固定，则应返回 NULL。

FlushPgImp 应该刷新一个页面而不管它是pin 还是unpin

UnpinPgImp，is_dirty 参数跟踪页面在固定时是否被修改。

上述函数所需要的一些具体的实现在你即将实现的代码中做了明确的需求，可以进行查看。

Pin 和 Unpin 在 LRUReplacer 和 BufferpoolManagerInstance的上下文中具有相反的含义，在 LRUReplacer 的上下文中，pin一个页面意味着我们不应该在内存中驱逐该页面，因为它正在使用。这意味着我们应该从 LRUReplace 中移除它。另一方面，在 BufferPoolManagerInstance 中pin一个页面意味着我们希望使用一个页面，并且不应该从缓冲池中删除该页面。

TASK #3 - PARALLEL BUFFER POOL MANAGER

单个缓冲池管理器实例需要采用锁存器，以保证线程安全。而当每个线程在与缓冲池进行交互时，都会争用一个锁存器，这会导致大量的争用。一个可能的解决方案是在系统中有多个缓冲池，每个缓冲池都有自己的锁存器。

ParallelBufferPoolManager 是一个包含多个 BufferPoolManagerInstance 的类。对于每个操作，ParallelBufferPoolManager 选择一个 BufferPoolManagerInstance 并委托给该实例。

我们使用给定的page id 来确定要使用哪个特定的 BufferPoolManagerInstance。如果我们有 num_ instance个 BufferPoolManagerInstance，那么我们需要一些方法来将给定的page id 映射到范围[0，num_ instance)中的一个数字。对于这个项目，我们将使用模运算符，page_id mod num_instance 将把给定的 page _ id 映射到正确的范围。

当第一次实例化 ParallelBufferPoolManager 时，它的起始索引应该为0。每次创建一个新页面时，您将尝试每个 BufferPoolManagerInstance，从起始索引开始，直到一个成功。然后将起始索引增加一。

确保在创建单独的 BufferPoolManagerInstances 时使用接受 uint32_t num_instance 和 uint32_t instance_index 的构造函数，以便正确创建pages id。

  1. 关于 page 和 frame

  • 数据在磁盘上以文件形式存储，数据库的存储引擎通常以固定大小的page(有的数据库称为block)为单位组织文件，读写磁盘均以page为单位。

  • 数据库在内存的buffer pool便是用来缓存磁盘中读取上来的page。buffer pool 同样以page大小管理组织内存块，这样一块内存被称为frame。

  • 当DBMS需要时，会将磁盘中的page拷贝到一个frame中或者将frame写入到磁盘中的page去。

TASK #1 - LRU REPLACEMENT POLICY

这部分主要实现LRU替换算法，但该要求与传统的方法不同。

数据结构

算法实现

  1. 构造函数，设定该LRUReplacer大小的容量
  1. Victim 函数，选择一个受害者（就是最近最久未使用的）
  1. Pin 函数，当进程执行锁定该页面时，调用该函数，需要删除该页面，使其不能称为LRU算法的对象。
  1. Unpin 函数，当进程使用完毕后释放此页面，调用该函数，需要增加该页面，使其可以成为LRU算法的对象。
  1. Size 函数，返回当前 LRU_Replacer 的大小

一些错误

  1. 为了保证线程安全，使用锁。std::lock_guard<std::mutex> lock(latch_);
  2. Unpin()这个方法只有当页面的pin_count变成0后才会被调用，这个思路在Task2中的UnpinPgImp会有体现。但值得注意的一点是，只有内存中不含此 frame_id 时，才会执行完整逻辑。如果含有此页面，并不会将此页面调用到最近的一页中，只会进行单纯的判断页面数目大小会不会超过其容量，超过了则删掉最久没用的，并把其插入到最新的一页。
  3. 在查找时，多多利用数据结构设计的unordered_map，不然在进行分数检测时，可能会报 timeout。

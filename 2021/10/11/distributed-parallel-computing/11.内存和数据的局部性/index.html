GPU的性能如何?

CGMA比率(计算与全局内存访问比率):在CUDA内核的一个区域内，每次访问全局内存所执行的浮点计算的数量。 越高越好

基本的矩阵乘法的CGMA： 所有线程因其输入矩阵元素访问全局内存。

  • 每次浮点加法一次内存访问（4 字节）Pvalue+=***的加法
  • 每次浮点乘法一次内存访问M[**]*N[***]的乘法
  • CGMA=1,每次迭代进行两次访存，两次浮点数运算。尽管最后还有一次的赋值语句需要访存，但相比之下，就可忽略不计了。
  • 4B/s 的内存带宽/FLOPS

假设一个GPU

  • 峰值浮点速率 1,500 GFLOPS，200 GB/s DRAM 带宽
  • 4* 1500 = 6000gb /s需要达到峰值FLOPS额定
  • 200 GB/s 的内存带宽将执行速度限制在 50 GFLOPS（200GB/S除以4B/s）

这限制了设备的峰值浮点执行率，到达了3.3% (50/1500) 需要大幅减少内存访问以接近 1,500 GFLOPS 为了达到顶峰 1,500 GFLOPS, 我们 需要 CGMA=30

CUDA中的共享内存

一种特殊类型的内存，其内容在内核源代码中明确定义和使用

  • 每个SM一个
  • 以比全局内存高得多的速度（在延迟和吞吐量方面）访问
  • 访问和共享范围
  • Lifetime——线程阻塞，对应线程结束执行后内容会消失
  • 通过内存加载/存储指令访问
  • 一种形式暂存器

SPM 体积较小、功耗低、访问速度比Cache 快。（快在哪？统一编址？） 程序员可以灵活控制是SPM的最大优势。 SPM由于和主存统一编址，处理器可以直接访问，而不会出现像 Cache 一样不命中的现象，也不会访问主存。

平铺/阻塞-基本概念

将全局内存内容划分为tiles（块） 在每个时间点将线程的计算集中在一个或少量的块上

平铺的基本概念

在拥堵的交通系统中，显着减少车辆可以大大改善所有车辆看到的延迟。

  • 拼车上下班
  • 平铺全局内存访问
    • 司机：访问其内存数据操作数的线程
    • 汽车：内存访问请求

一些计算对平铺更具挑战性

  • 有些拼车可能比其他拼车更容易
    • 拼车参与者需要有类似的工作时间表
    • 有些车辆可能更适合拼车
  • 平铺也存在类似的挑战

需要同步

当人们有相似的日程安排时是很好的 但当人们的日常安排不同时，就会很差。

和平铺一样：

好的: 当线程具有相似的访问时序时 坏的：当线程有非常不同的时间时

平铺的技术概要

1.识别由多个线程访问的全局内存内容块 2.将 tile 从全局内存加载到（on-chip）共享内存中 3.用 障碍 同步以确保所有线程都准备好开始阶段 4.让多个线程从共享存储器访问它们的数据 5.用 障碍 同步以确保所有线程都已完成当前阶段 6.移动到下一个块

块的矩阵乘法内核

数据访问模式：

  • 每个线程-M矩阵的一行，N矩阵的一列
  • 每个线程块-M矩阵的一条，N矩阵的一条

平铺矩阵乘法：

  • 将每个线程的执行分解成阶段
  • 这样每个阶段的线程块的数据访问都集中在M的一个块和N的一个块上
  • 块是每个维度中的BLOCK SIZE元素

加载图块

一个块中的所有线程都参与，每个线程在块中代码中加载一个 M 元素和一个 N 元素

共享内存允许多个线程访问每个值

障碍同步

同步块中的所有线程：__syncthreads() 同一个块中的所有线程必须到达 __syncthreads()才能继续前进 最适合用于协调分阶段执行平铺算法

  • 确保在阶段开始时加载图块的所有元素
  • 确保在阶段结束时消耗图块的所有元素

处理块的边界条件

加载M,N的输入图块0（Phase 0）

让每个线程在其P元素相同的相对位置加载一个M元素和一个N元素。

用于访问Tile 0的2维索引:M[Row][tx],N[ty][Col]

加载N的输入图块1（Phase 1）

用于访问Tile 0的2维索引:M[Row][1*TILE_WIDTH + tx],N[1*TILE*WIDTH + ty][Col]

平铺线程块大小注意事项

每个 线程块 应该有很多线程 1.16 的 TILE_WIDTH 给出 16* 16 = 256 个线程

对于 16，在每个阶段，每个块从全局内存执行 2*256 = 512 次浮点加载，为256 * (2*16) = 8,192 乘法/加法操作。（每次内存加载 16 次浮点运算）

CGMA=16，浮点加载的次数2（M,ds_M）*256(tx，ty的循环加载16x16) 浮点加法/乘法运算256(ds_M[*][ty]*ds_N[*****][tx])，*2（Pvalue+=...和ds_M*ds_N）的浮点数运算，*16(TILE_WIDTH)

2.32 的 TILE_WIDTH 给出 32* 32 = 1024个线程

对于 32，在每个阶段，每个块从全局内存执行 2*1024 = 2048 次浮点加载，为 1024 * (2*32)= 65,536 乘法/加法操作。（每次内存加载32个浮点运算）

CGMA=32,分析方法和上面一样。

共享内存和线程

对于具有 16KB 共享内存的 SM

共享内存大小取决于实现！ 对于 TILE_WIDTH = 16，每个线程块使用2*256*4B = 2KB 的共享内存。 对于 16KB 共享内存，最多可能有 8 个线程块在执行 这允许多达8*512 = 4,096 个待处理负载。（每个线程 2 个，每个块 256 个线程）

对于具有 32KB 共享内存的 SM

但 TILE_WIDTH 32 将导致每个线程块使用 2*32*32*4B= 8KB 共享内存，从而允许 2 个线程块同时处于活动状态 。 但是，当前一代 GPU 中每个 SM 1536 个线程的线程数限制将使每个 SM 的块数减少到一个！ 每个__syncthread()可以减少一个块的活动线程数 更多的线程块可能是有利的

任意矩阵维数的平铺核

处理任意大小的矩阵

到目前为止，我们介绍的平铺矩阵乘法内核只能处理维度（宽度）是平铺宽度（TILE_WIDTH）倍数的方阵。

  • 但是，实际应用程序需要处理任意大小的矩阵。
  • 可以将行和列填充（添加元素）为切片大小的倍数，但会产生大量的空间和数据传输时间开销。

我们将采取不同的方法

案例中的主要情况

线程不计算有效 P 元素但仍需要参与加载输入块

  • 块（1，1），线程（1，0）的第0阶段，分配计算不存在的P[3，2]，但需要参与加载块元素N[1，2]

计算有效 P 元素的线程可能会在加载输入块时尝试加载不存在的输入元素

  • 块（0，0）、线程（1，0）的阶段0被分配来计算有效的P[1，0]，但尝试加载不存在的N[3，0]

简单的解决方法

当一个线程要加载任何输入元素时，测试它是否在有效的索引范围内

  • 如果有效，继续加载
  • 否则，不加载，只写一个 0

基本原理：0 值将确保乘加步骤不会影响输出元素的最终值 加载输入元素的测试条件与计算输出 P 元素的测试不同

  • 不计算有效 P 元素的线程仍然可以参与加载输入 tile 元素

边界条件

输入M块的边界条件

每个线程加载：

  • M[Row][p*TILE_WIDTH+tx]
  • M[Row*Width + p*TILE_WIDTH+tx]

需要测试：

  • (Row < Width) && (p*TILE_WIDTH+tx < Width)
  • If true, load M element，Else , load 0

输入N块的边界条件

每个线程加载：

  • N[p*TILE_WIDTH+ty][Col]
  • N[(p*TILE_WIDTH+ty)*Width+ Col]

需要测试：

  • (p*TILE_WIDTH+ty < Width) && (Col< Width)
  • If true, load N element，Else , load 0

处理一般矩形矩阵

一般来说，矩阵乘法是按照矩形矩阵来定义的

  • j x k M矩阵乘以k x lN矩阵导致j x lP矩阵

我们提出了方矩阵乘法，一个特殊情况 需要通用内核函数来处理一般矩形矩阵

  • 宽度参数被三个参数所取代：j、k、l
  • 当宽度用于指P的M或高度时，将其替换为j
  • 当宽度用于指 N 的 M 宽度或高度时，请将其替换为 k
  • 当宽度用于指P的N宽或宽度时，请将其替换为l
